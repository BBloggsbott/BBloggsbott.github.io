You type a message into a chat window and an AI agent responds back. But you're not actually "chatting" with an LLM. Generative LLMs are essentially extremely powerful autocomplete engines. When we build an AI agent, we just have it call the [`/v1/chat/completions`](https://developers.openai.com/api/reference/resources/chat/subresources/completions/methods/create) with info about the available tools, the response format, and other parameters. But in the early days of Generative LLMs, we didn't have an endpoint specifically designed to handle conversations. We only had the [`/v1/completions`](https://developers.openai.com/api/docs/guides/completions) endpoint, which is now a legacy endpoint in OpenAI. So how do these powerful autocomplete engines have conversations and how are we able to influence the response of these models with the params we send as part of the request?

*__Note:__ A lot of these findings are based on some experiments and research I did with models available on huggingface like Llama and Qwen. So even though the core idea remains the same, the implementation details might vary depending on the model you use*

## Chat Completion vs Regular Completion
Generative LLMs are text compeltion engines. Given a sequence of text, these models are great at predicting the next sequence of tokens. Developers would use structured prompts to take advantage of the text completion capabilities of the model to mimic chat conversations before we had chat specific completion endpoints. This is what a sample prompt to the Completions endpoint looked like back then:
```
The following is a conversation with an AI assistant. The assistant is helpful, creative, clever, and very friendly

Human: Hello, who are you?
AI:
```
The LLM would then generate the best sequence of text to follow the prompt, and that was considered to be the model's reponse.


The Chat completion endpoints are wrappers that similarly format the messages before sending them to the completion engine.

## The Need for a Translation Layer
Our API calls to the LLM servers are JSON objects that contain the messages in the conversation and several other parameters that help control the text generated by the model. JSON is very simple, easy to read and understand. Why don't we just pass it to the model have it generate predictions based on that? For example, why don't we just use a prompt as shown below?
```
[
    {
        "role": "system",
        "content": "You are a helpful assistant" 
    },
    {
        "role": "user",
        "content": "Hello" 
    }
]

For this conversation, the AI assistant's reply would be:
```

This might work. But it is not the best way because
1. **Transformers only understand 1D Sequences** - LLMs do not understand data structures. They do not know what a "dictionary" or an "array" or a "key-value pair" is. If you feed a JSON object directly into a model, it doesn't parse it like a computer program does. It just reads it as a long sequence of characters: curly braces, quote marks, spaces, colons, and letters.
2. **The Verbosity Tax** - When using LLMs, every token costs money, and JSON is incredibly verbose. In the above example, consider the user message. To a tokenizer, that string includes tokens for `{`, `"`, `role`, `"`, `:`, `"`, `user`, `"`, `,`, `"`, `content`, `"`, `:`, `"`, `Hello`, `"`, `}`. That is a massive amount of "syntactic noise" just to pass the word "Hello."
3. Before an LLM is fine-tuned to act like a chatbot, it is a "base model" pre-trained on trillions of words of raw, unstructured human text (books, articles, websites). Because their core nature is to continue raw text streams, it is vastly easier and more performant to fine-tune them using a lightweight text-markup format (like Jinja chat templates) rather than forcing them to learn strict JSON syntax just to hold a conversation.

## The Translation Layer

To simplify the JSON into something the model can easily understand and keep the cost low, developers were using simple prefixes and suffixes to build the actual text that goes to the LLM. For example, the sample messages shown in the previous section would look like this:
```
<|system|>You are a helpful assistant<|end|>
<|user|>Hello<|end|>
<|assistant|>
```
The model will then complete this sequence of text with the assistant's reponse and the generation would stop when the token `<|end|>`. Models could be finetuned to strictly follow this structure whenever the model sees it. The special terms you see here like `<|end|>`, `<|system|>` and `<|user|>` would be treated as one token each because of how the tokenizers were built (read more about it [here]({{ base_path }}/read-a-paper/gpt-2/#byte-pair-encoding)).

This worked fine. However, as the models became more complex, their input structure also started getting more complex. Some models needed whitespaces in specific parts of the inputs. Some models needed conditional logics and some needed loops to parse things like json with tool information. They could ship some python code with the model which takes care of this. But this will not work if you need to easily port your model to work on other languages. It also requires executing untrusted python code in your machine, which is a massive security risk.

To solve this problem, Chat Templates build on [Jinja](https://jinja.palletsprojects.com/en/stable/) were introduced.

### Jinja Templates

>Jinja is a fast, expressive, extensible templating engine. Special placeholders in the template allow writing code similar to Python syntax. Then the template is passed data to render the final document.

This allows Jinja to solve all the problems in the older approach. 
* Jinja, essentially being a mini programming language enabled conditional templates and looping on obejcts to build extensive strings.
* Jinja templates are plaintext strings, which can be safely stored and processed with the model. 
* Jinja might have started in python, but Jinja template parsers are available in almost every programming language.

Almost all models available on Huggingface will have the chat template along with the model. For example, you can find `Llama-3.2-3B-Instruct`'s chat template [here](https://huggingface.co/meta-llama/Llama-3.2-3B-Instruct/blob/main/tokenizer_config.json#L2053).

Since the last token of the prompt is `<think>`, it will continue generating text to reason perform reasoning for the user's query, generate a `</think>` token to indicate that it is done thinking. The tokens generate after the `</think>` tag are to be treated as the response to the user's query. `<think></think>` essentially gives the model a scratchpad to write down it thoughts while solving a problem.

## The JSON Schema Magic

LLM have a hard time generating structured outputs. In the earlier days of Generative LLMs, getting a valid JSON with the expected keys from the model felt like a miracle. But now, a lot of chat completions endpoint allow defining the response schema with ensures that the response structly adheres to the given structure. The two widely used ways to achieve this is using Grammars or Outlines.

### Grammars
A Grammar is a formal set of rules that defines the structure of a language. It dictates how individual elements (like words or characters) can be combined to form valid "sentences" or strings. Model Servers like llama.cpp support the use of GBNF (Grammar Based Native Form) files to define a formal grammar for speciic structured outputs like [`json`](https://github.com/ggml-org/llama.cpp/blob/master/grammars/json.gbnf) or [`chess`](https://github.com/ggml-org/llama.cpp/blob/master/grammars/chess.gbnf).

Normally, an LLM predicts the next most likely token from its entire vocabulary. Without constraints, it might hallucinate a comma where a bracket should be in a JSON object. GBNF changes the Sampling process:
* Parsing: The llama.cpp engine tracks the "state" of the grammar as it generates text.
* Logit Masking: Before the model picks the next word, the grammar engine "masks" (sets the probability to zero) for any tokens that would violate the grammar rules.
* Guaranteed Validity: Because the model is physically prevented from choosing an illegal token, the output is guaranteed to be syntactically correct.

### Outlines
[Outlines](https://github.com/dottxt-ai/outlines/tree/main) is a library that also guarantees structure in an LLM's response. It does this using a more mathematically optimized approach called [Finite State Machines](https://en.wikipedia.org/wiki/Finite-state_machine). Just like GBNF, Outlines operates at the Logit level. Before the model picks the next word, Outlines looks at all possible tokens in the model's vocabulary (often 32,000+ tokens) and sets the probability of "illegal" tokens to negative infinity ($$-\infinity$$). The model literally cannot see the wrong characters. It only "sees" the tokens that fit your structure.

The biggest difference between Outlines and other tools is how it decides which tokens are legal.
* **Conversion to Regex**: Whether you provide a Pydantic model, a JSON schema, or a list of types, Outlines converts that requirement into a Regular Expression.
* **Regex to FSM**: It then compiles that regex into a Deterministic Finite Automaton (DFA) or FSM.
* **Token Mapping**: This is the clever part. Outlines pre-indexes the model's entire vocabulary against the FSM. For every state in the FSM, Outlines knows exactly which tokens in the model's brain will trigger a transition to a valid next state.


It is incredibly easy to treat modern LLMs as magical black boxes. You send a message, and the "AI" texts you back.

But as we've explored, you are not chatting with an AI. You are passing stateless data to a text-prediction matrix through a beautifully orchestrated, highly vulnerable translation layer. The next time you type `{"role": "user", "content": "Hello"}` into your code editor, you'll know exactly what is happening beneath the surface. The chat interface is just an illusion, but knowing how the magic trick works makes you a vastly better magician.

## Footnote {#footnote}
### Why JSON for tools?
When OpenAI first introduced function calling, they decided to use JSON Schema because it's the web standard for APIs. Because OpenAI set this standard, the rest of the industry followed suit. When Meta trained Llama 3 or Alibaba trained Qwen, they explicitly fine-tuned their models on massive datasets featuring JSON schemas paired with function calls. To the model, JSON Schema is no longer an arbitrary, complex data structure—it is a highly specific "dialect" of text that it has been rigorously trained to read.

The observation that JSON is too complex/wasteful for tools is actually a massive topic in the AI engineering community right now. Because JSON schemas eat up so many valuable tokens, advanced developers and newer frameworks are starting to abandon JSON entirely in their chat templates. Instead, they write scripts that convert the user's JSON API request into TypeScript Interfaces or YAML before injecting it into the template.

Look at the difference
#### JSON Schema
```json
"parameters": {
  "type": "object",
  "properties": {
    "location": { "type": "string" }
  }
}
```

#### Typescript Interface
```typescript
interface get_weather {
  location: string;
}
```

 Qwen also uses a different response structure in the tool call response. The `<tool_call>` and `</tool_call>` are treated as a single token in the output.
